AutoBudget

A full-stack personal finance platform with bank integration, AI categorization, and Slack-driven budget management.

Description

Most budgeting apps share the same fundamental problem: people don't open them. You set up categories, link your accounts, and then never look at the app again until you've already overspent. AutoBudget was built around a different premise — instead of waiting for you to check in, it comes to you. Every time a transaction hits your bank account, you get an instant Slack message showing what you spent, where it was categorized, and exactly how much of that category's budget remains. This real-time awareness helps you make proactive spending decisions in the moment, not after the fact.

At the end of each month, a Slack-driven reconciliation flow holds you accountable — walking through every category where you went over budget and requiring a decision about where those funds come from, while incentivizing savings by letting you see surplus accumulate. The result is a budgeting system that stays in your face just enough to actually change behavior.

Under the hood, AutoBudget is a full-stack TypeScript platform. Users connect their bank accounts via Plaid, transactions are automatically synced and categorized using AI, and spending is tracked against a custom budget — all without manual data entry.

The entire stack is written in TypeScript — Angular 20+ on the frontend and Express.js on the backend, with PostgreSQL as the database. This decision was deliberate: shared type definitions between the frontend and backend eliminate entire classes of runtime errors and make refactoring safe. For database access, Drizzle ORM was chosen over heavier alternatives like TypeORM or Sequelize because it generates type-safe query builders directly from the schema definition, with minimal runtime overhead and readable SQL output that is easy to debug. The frontend is deployed on Firebase Hosting for global CDN distribution, while the backend and database run on Railway for streamlined deployment with automatic migrations on startup.

Plaid Integration & Transaction Sync

Bank account connectivity is powered by Plaid's API through a multi-step secure handshake. The backend generates a link token, the user authenticates through Plaid's embedded Link UI, and the resulting public token is exchanged server-side for a permanent access token that is immediately encrypted before storage.

Transaction syncing is webhook-driven rather than polling-based. When new transactions are available, Plaid sends a webhook that triggers a cursor-based sync — each sync picks up exactly where the last one left off, avoiding duplicate fetches. The webhook verification pipeline is multi-layered:

  • JWT Signature Verification — The webhook payload is a signed JWT. The server fetches Plaid's public verification key and validates the signature before processing.
  • Body Hash Validation — A SHA-256 hash of the raw request body is compared against the hash embedded in the JWT, ensuring the payload hasn't been tampered with in transit.
  • Timestamp Check — Webhooks older than 5 minutes are rejected to prevent replay attacks.

A critical design decision here is that the server responds with HTTP 200 immediately upon receiving the webhook, before any processing begins. Plaid requires responses within 5 seconds, and transaction processing — which includes AI categorization and Slack notifications — can take significantly longer. The actual processing happens asynchronously after acknowledgment, with errors stored in the database for inspection and retry rather than surfaced to Plaid. Individual transaction failures don't block the rest of the batch.

AI-Powered Transaction Categorization

Every incoming transaction is categorized automatically using OpenAI's GPT-3.5-turbo. Rather than sending just the transaction name and amount, the prompt is enriched with multiple layers of context to maximize accuracy:

  • Transaction Metadata — Amount, merchant name, Plaid's own category (when available), and the transaction description.
  • User's Budget Categories — The full set of categories and subcategories from the user's budget, so the AI assigns to valid destinations.
  • Merchant History — The last 3 transactions from the same vendor and how they were categorized, biasing the model toward consistency.
  • Manual Override History — When a user corrects a categorization, that override is stored in a dedicated table and included in future prompts for the same merchant, allowing the system to learn from corrections over time.

Before hitting the LLM at all, a transfer detection layer checks for keywords (TRANSFER, PAYMENT, CREDIT CARD, AUTOPAY) and Plaid category patterns (TRANSFER_IN, LOAN_PAYMENTS, etc.) to automatically classify obvious non-spending transactions as "Excluded" — keeping them out of the budget without wasting an API call.

Security Architecture

Sensitive third-party tokens — Plaid access tokens and Slack OAuth tokens — are encrypted at rest using AES-256-GCM, an authenticated encryption scheme that provides both confidentiality and integrity verification. Each encryption uses a unique random IV and salt, so the same token never produces the same ciphertext. The encryption key is derived from an environment variable, keeping it fully separate from the database. Backward compatibility logic detects legacy unencrypted tokens and handles them gracefully during migration.

Webhook endpoints verify the authenticity of incoming requests from both Plaid and Slack using their respective signature schemes, with timestamp validation to prevent replay attacks. User authentication is JWT-based with bcrypt password hashing, and Helmet.js enforces standard HTTP security headers across the API.

Slack-Guided Month-End Reconciliation

At the end of each month, the system needs to reconcile actual spending against budgeted amounts. Fixed categories (like rent or subscriptions) are processed automatically — surplus rolls over or deficit is noted. But variable categories (groceries, dining, entertainment) require user decisions about what to do with leftover or overspent funds. Rather than building a custom UI for this, the reconciliation flow is driven entirely through Slack.

The flow presents one category at a time via interactive Slack messages with buttons. For categories with surplus, the user chooses where to move the extra funds. For deficits, they choose which category to pull from. Partial coverage is supported — a message updates in place as the user allocates funds across multiple destinations. A final "Lock Month" button creates immutable snapshots of every category's end state.

Real-time transaction notifications are also delivered through Slack, showing the category, merchant, amount, and a budget progress indicator. Users can correct categorizations directly from the notification via buttons, or initiate a split transaction through a two-step modal flow: first selecting the number of splits, then filling in amounts for each — with the last split auto-calculating the remaining balance and validation ensuring the splits sum to the original transaction amount.

The entire flow is tracked by a state machine (Pending, In Progress, Awaiting User Input, Completed) with every fund movement recorded in a dedicated table for a complete audit trail.

Database Design

The database schema is defined in TypeScript using Drizzle ORM's schema-first approach. The schema file serves as the single source of truth — running drizzle-kit generate produces SQL migration files automatically, and migrations run on every server startup, keeping the database always in sync with the code without manual deployment steps.

The data model is organized around several key relationships:

  • Budget Hierarchy — Users own budgets, budgets contain categories, and categories contain subcategories. A unique constraint enforces one budget per user, simplifying all business logic to a single source of truth.
  • Banking Data — Plaid items (bank connections) own accounts, and accounts own transactions. Transaction IDs from Plaid are stored with a unique constraint to prevent duplicates during re-syncs.
  • Cascading Deletes — Deleting a user cascades through all related data (budgets, categories, transactions, Plaid connections), maintaining referential integrity without application-level cleanup logic.
  • Flexible Storage — Plaid's category arrays are stored as JSON columns rather than normalized tables, preserving the third-party data structure while remaining queryable with PostgreSQL's JSON operators.
  • Month-End Auditing — Fund movements and monthly snapshots are stored in separate tables, creating an immutable audit trail of every reconciliation decision.
