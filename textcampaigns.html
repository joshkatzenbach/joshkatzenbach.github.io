Text Campaigns

Automated messaging platform for sales teams — built at Enzy.

Description

At Enzy, I helped build the text campaign system — an automated messaging platform that enables sales organizations to run targeted, data-driven messaging programs at scale. Managers configure campaigns tied to live performance metrics, and the system automatically sends the right message to the right person at the right time based on their actual numbers.

The platform supports rich message types beyond plain text — images, videos, documents, GIFs, multi-image attachments, embedded reports, polls, and more. It integrates with Twilio for SMS delivery and handles inbound responses, connecting them back to surveys and forms within the app. The entire system is multi-tenant, isolating data at the workspace level so multiple organizations run on the same infrastructure without overlap.

The backend is Java with a DAO-based persistence layer on AWS Aurora MySQL, and the frontend is an Angular/Ionic hybrid mobile application. Media assets are stored across dedicated S3 buckets with a normalized URL storage pattern that eliminates duplication.

Note: This was built as part of my work at Enzy. Source code is proprietary and not publicly available.

Messaging Program Architecture

The core abstraction is the "messaging program" — a configurable campaign that ties message delivery to report data. Each program defines a date range, a set of trigger conditions, and the content to send when those conditions are met. Triggers map to specific metric values from live reports, so a manager can set up rules like "send this motivational video to anyone who hits 80% of their monthly target" or "send a reminder to reps with zero activity this week."

Each trigger stores its own message content and type — a trigger might send a plain text message, while another sends a video with a thumbnail, and a third sends an image. This flexibility means a single program can deliver different content to different segments of the team based on where they stand, all driven by real performance data rather than static distribution lists.

Scheduled Execution

Campaign execution is handled by a Quartz-based job scheduler with time-window constraints. Programs only fire during configurable business hours, preventing messages from going out at inappropriate times. The job iterates through every active workspace, finds programs that fall within their date range and haven't already run, loads the relevant report data, groups users by their trigger values, and dispatches messages accordingly.

The job implements an interruptible interface for graceful server shutdown — if the server needs to restart mid-execution, in-flight work stops cleanly rather than leaving the system in an inconsistent state. Execution timestamps are tracked per program so the scheduler knows exactly what has and hasn't been processed.

Message Delivery & Rich Media

The message system supports a wide range of content types through a type enum that drives type-specific processing throughout the stack. When a message is sent, the system:

  • Validates permissions — Checks conversation membership, admin-only channel restrictions, and deduplicates by frontend ID to prevent double-sends.
  • Handles media — Images, videos, and documents are uploaded to dedicated S3 buckets. Videos go through an AWS MediaConvert transcoding pipeline. Multi-image messages are stored via a separate attachment table.
  • Extracts link previews — URL messages automatically generate and store preview metadata.
  • Dispatches notifications — Real-time socket notifications for in-app delivery, push notifications for native mobile, or both depending on message type. Video messages use socket-only delivery until transcoding completes.

Twilio SMS Integration

The platform integrates with Twilio for outbound SMS delivery and receives inbound responses via a webhook endpoint. When a recipient replies to an SMS, Twilio posts the message along with metadata — sender, geographic information, media count — to the webhook receiver. The system persists the response, matches it back to the original question recipient, and links it into the survey/form system so managers can see responses alongside campaign performance.

Multi-Tenant Data Architecture

Every table in the system includes workspace and company identifiers, enforcing tenant isolation at the database level. This means multiple sales organizations share the same infrastructure while their data remains completely separated — campaigns, contacts, messages, and analytics are all scoped to the workspace.

Media storage uses a normalized S3 URL pattern — rather than storing full URLs on every record, URLs are stored once in a dedicated table and referenced by ID. This reduces data duplication across the millions of messages and attachments in the system and makes it straightforward to update CDN paths or migrate storage without touching every record. Soft deletes with timestamps are used throughout, preserving audit trails and enabling data recovery.
