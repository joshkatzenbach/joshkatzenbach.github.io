SalesMind Analytics

AI-powered sales call analysis platform for coaching and performance improvement.

Description

Sales coaching has traditionally been a manual, time-intensive process — managers listen to recorded calls, take notes, and try to identify patterns across their team. SalesMind Analytics was built to automate that entire workflow. Upload a sales call transcript, and the platform analyzes it against customizable skill frameworks, extracts and categorizes objections, scores overall performance, and surfaces actionable coaching insights — all powered by AI.

The platform is designed for sales teams who want data-driven coaching at scale. Rather than relying on a manager's subjective impression of a call, SalesMind produces structured assessments: how well did the rep handle discovery? Were objections addressed effectively? What skill areas need development? Each call can be evaluated against multiple selling methodologies simultaneously, giving teams flexibility in how they measure performance.

The application is a full-stack TypeScript monorepo — Angular 21 on the frontend, Express.js on the backend, and PostgreSQL for the database. An npm workspaces monorepo structure with a shared types package ensures type safety across the entire stack, eliminating API contract mismatches between frontend and backend. The frontend deploys to Firebase Hosting and the backend runs on Railway.

Note: This is a proprietary product. Source code is not publicly available.

Analysis Pipeline & Queue Architecture

The core challenge in building the analysis pipeline was handling API rate limits. A single transcript analysis involves multiple LLM calls — skill evaluation, objection extraction, and categorization — and processing several transcripts concurrently would quickly overwhelm the API.

The solution is a database-backed job queue with globally serialized processing. Uploads return immediately and enqueue the analysis job in the background. A processing loop picks up one job at a time, completes it fully, then moves to the next. This design prioritizes reliability over speed — rather than parallelizing aggressively and building complex retry logic, serialization with intelligent recovery proved far more dependable. On server restart, any in-flight analyses are automatically reset to pending, preventing stuck jobs.

Skill Framework Analysis

A key architectural decision was making skill frameworks user-configurable rather than hard-coded. Each user can create custom frameworks with their own skills and proficiency levels (e.g., "Emerging," "Proficient," "Expert"). When a transcript is analyzed, the AI extracts skill scores and generates encounter metadata (title, date, deal status), then a separate two-step pipeline handles objection extraction and categorization. A normalized score across all evaluated skills produces the overall encounter score.

Objection Extraction & Categorization

Objection handling is split into two distinct AI calls rather than one monolithic prompt, and this separation was a deliberate design decision. The first call extracts raw objections from the transcript — what was the objection, how did the rep respond, was it resolved, and which was the primary objection. This extraction is completely independent of any categorization system.

The second call takes those extracted objections and classifies them along two axes: type (e.g., "Price/Budget," "Implementation Timeline") and category/subcategory (e.g., "Financial," "Technical"). Types are dynamic — the AI can create new ones when existing types don't fit. Categories are curated through an admin interface.

The reason for separating extraction from categorization is evolution. As the category taxonomy changes over time — new types emerge, categories get reorganized — old transcripts don't need to be re-analyzed. The raw objection data stands on its own. Only the categorization layer adapts to the evolving taxonomy.

Cost Tracking & Resilience

Every LLM call is logged as a first-class entity in the database — token counts, model, request type, and computed costs based on dynamic pricing. This enables a cost dashboard that breaks down AI spending by request type and per-encounter, providing transparency for pricing decisions.

All AI calls are wrapped in exponential backoff with jitter to handle transient failures gracefully. Transcript text is stored at upload but cleared after successful analysis to minimize storage — if analysis fails, the text is retained so the user can retry without re-uploading.

Pipeline A/B Testing Framework

As the platform evolves, a key question is: how do you know if a new AI prompt, model, or analysis strategy actually produces better results? To answer this, the system includes a dedicated A/B testing framework for comparing different AI pipelines side by side.

The architecture is built around a pluggable pipeline registry. Each pipeline implements a standardized interface — it takes in a transcript and a skill framework, and returns scores, objections, and metadata. What happens inside is entirely up to the pipeline: it could use a different model, a different prompt strategy, chain-of-thought reasoning, or any other approach. New pipelines are added by creating a single file and registering it — no changes to core analysis code required.

Users upload test transcripts, select which pipelines to run, and view results in a comparison grid — skills as rows, pipelines as columns. Manual reference scores can be entered alongside the AI results, making it easy to see which pipeline most closely matches human judgment. Test encounters are flagged in the database so they never pollute production analytics, and deduplication logic prevents redundant runs of the same transcript-pipeline-framework combination.

Integration System

External transcript sources connect through a webhook adapter pattern, making it straightforward to add new integrations as they arise. Current adapters support both simple JSON payloads (for tools like Zapier) and platform-specific integrations with signature verification (like Fathom for video calls). API keys are hashed before storage and shown only once at creation.

Database Design

The database evolved through 14 migrations, reflecting iterative feature development. The schema is built around several key entity groups:

  • Encounters & Analysis — Sales calls with metadata, linked to a database-backed analysis queue for async processing.
  • Skill Frameworks — User-defined frameworks containing skills with configurable proficiency levels. Per-encounter skill scores link calls to framework evaluations.
  • Objection Taxonomy — A dual-axis classification system with dynamic types and curated categories/subcategories, allowing the taxonomy to evolve independently of extracted data.
  • Cost Tracking — Per-request LLM logging with dynamic pricing lookup, enabling precision cost analytics without post-aggregation.
  • Integrations — API key storage and webhook audit logs for external transcript ingestion.

Cascade deletes on encounters automatically clean up skill analyses and objections. The analyzed flag ensures statistics queries only count complete analyses, and transcript text is cleared after successful processing to minimize storage.
