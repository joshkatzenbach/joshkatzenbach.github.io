SalesMind Analytics

AI-powered sales call analysis platform for coaching and performance improvement.

Description

Sales coaching has traditionally been a manual, time-intensive process — managers listen to recorded calls, take notes, and try to identify patterns across their team. SalesMind Analytics was built to automate that entire workflow. Upload a sales call transcript, and the platform analyzes it against customizable skill frameworks, extracts and categorizes objections, scores overall performance, and surfaces actionable coaching insights — all powered by AI.

The platform is designed for sales teams who want data-driven coaching at scale. Rather than relying on a manager's subjective impression of a call, SalesMind produces structured assessments: how well did the rep handle discovery? Were objections addressed effectively? What skill areas need development? Each call can be evaluated against multiple selling methodologies simultaneously, giving teams flexibility in how they measure performance.

The application is a full-stack TypeScript monorepo — Angular 21 on the frontend, Express.js on the backend, and PostgreSQL for the database. An npm workspaces monorepo structure with a shared types package ensures type safety across the entire stack, eliminating API contract mismatches between frontend and backend. The frontend deploys to Firebase Hosting and the backend runs on Railway.

Note: This is a proprietary product. Source code is not publicly available.

Analysis Pipeline & Queue Architecture

The core challenge in building the analysis pipeline was handling OpenAI's rate limits. A single transcript analysis involves multiple LLM calls — one per skill framework, plus a multi-step objection extraction pipeline — and processing several transcripts concurrently would quickly trigger 429 rate limit errors.

The solution is a database-backed job queue with globally serialized processing. When a user uploads a transcript, the server returns immediately with a placeholder encounter and enqueues the analysis job. A background loop picks up one job at a time, processes it fully, then moves to the next. The frontend polls for status updates every 60 seconds.

This design choice prioritizes reliability over speed. Rather than trying to parallelize AI calls aggressively and building complex retry logic for rate limit failures, serialization with intelligent recovery proved more dependable in practice. On server restart, any in-flight analyses are automatically reset to pending, preventing stuck jobs. The background loop uses setImmediate() recursion instead of timers, avoiding race conditions while maintaining non-blocking execution.

Multi-Framework Skill Analysis

A key architectural decision was making skill frameworks user-configurable rather than hard-coded. Each user can create custom frameworks with their own skills and proficiency levels (e.g., "Emerging," "Proficient," "Expert"). When a transcript is analyzed, it's evaluated against every framework the user has defined — meaning the same call can be scored against MEDDIC, Sandler, Challenger, or any custom methodology simultaneously.

The analysis flow for a single transcript works in stages:

  • First Framework — The AI extracts skill scores and generates encounter metadata (title, date, deal status). This first pass establishes the baseline context for the call.
  • Remaining Frameworks — Each additional framework is processed sequentially to stay within rate limits, but conceptually they're independent evaluations.
  • Objection Analysis — A separate two-step pipeline extracts objections from the transcript, then categorizes them (detailed in the next section).
  • Score Calculation — A normalized average across all frameworks produces the overall encounter score.

Objection Extraction & Categorization

Objection handling is split into two distinct AI calls rather than one monolithic prompt, and this separation was a deliberate design decision. The first call extracts raw objections from the transcript — what was the objection, how did the rep respond, was it resolved, and which was the primary objection. This extraction is completely independent of any categorization system.

The second call takes those extracted objections and classifies them along two axes: type (e.g., "Price/Budget," "Implementation Timeline") and category/subcategory (e.g., "Financial," "Technical"). Types are dynamic — the AI can create new ones when existing types don't fit. Categories are curated through an admin interface.

The reason for separating extraction from categorization is evolution. As the category taxonomy changes over time — new types emerge, categories get reorganized — old transcripts don't need to be re-analyzed. The raw objection data stands on its own. Only the categorization layer adapts to the evolving taxonomy.

LLM Cost Tracking

Every OpenAI API call is logged as a first-class entity in the database — not aggregated after the fact, but tracked at the individual request level with input/output token counts, model name, request type (transcript analysis, objection extraction, objection categorization, skill generation), and computed costs based on dynamic pricing stored in the database. This enables a cost dashboard that breaks down spending by request type and per-encounter, providing full transparency into AI spending and enabling data-driven pricing decisions.

Resilience Patterns

Every LLM call is wrapped in exponential backoff with jitter — up to 3 retries with delays that roughly double each time, plus random variance to prevent multiple failed requests from retrying simultaneously. Rate limit (429) and service unavailable (503) errors are retried; authentication and content filter errors are not.

Transcript text is stored on the encounter at upload time but cleared after successful analysis to free storage. If analysis fails, the text is retained so the user can retry without re-uploading. An analyzed flag gates all statistics queries, ensuring only fully completed analyses are counted in dashboards and reports.

Integration System

External transcript sources connect through a webhook adapter pattern. Two adapters are currently supported: a generic adapter for tools like Zapier (simple JSON payload) and a Fathom adapter for the video call platform (with HMAC-SHA256 signature verification). Each integration authenticates via API keys — prefixed with sm_live_, stored as SHA-256 hashes (the raw key is shown once at creation and never again), and accepted via either Bearer header or query parameter to accommodate platforms that can't set custom headers.

Database Design

The database evolved through 14 migrations, reflecting iterative feature development. The schema is built around several key entity groups:

  • Encounters & Analysis — Sales calls with metadata, linked to a database-backed analysis queue for async processing.
  • Skill Frameworks — User-defined frameworks containing skills with configurable proficiency levels. Per-encounter skill scores link calls to framework evaluations.
  • Objection Taxonomy — A dual-axis classification system with dynamic types and curated categories/subcategories, allowing the taxonomy to evolve independently of extracted data.
  • Cost Tracking — Per-request LLM logging with dynamic pricing lookup, enabling precision cost analytics without post-aggregation.
  • Integrations — API key storage and webhook audit logs for external transcript ingestion.

Cascade deletes on encounters automatically clean up skill analyses and objections. The analyzed flag ensures statistics queries only count complete analyses, and transcript text is cleared after successful processing to minimize storage.
